# 技术选型方案：AI化工信息化系统

## 1. 设计原则

1. **渐进增强**：传统业务系统（EHS/MES/ERP）作为能力后台，AI作为编排层，不推倒重来。
2. **国产优先**：化工行业对数据主权敏感，优先选择国内可私有化/本地化部署的大模型和基础设施。
3. **离线兜底**：现场网络不稳定，移动端必须具备离线缓存和弱网适配能力。
4. **可审计**：所有AI推理结果必须可溯源，满足化工行业合规要求。

## 2. 大模型选型

### 2.1 核心需求
- 中文语义理解能力强（化工术语、方言口音文本）
- 支持 Function Calling / Tool Use（调用设备台账、历史隐患等API）
- 支持私有化部署或国内合规云服务
- 成本可控（化工企业数据量大，调用频繁）

### 2.2 推荐方案：分层模型策略

| 场景 | 推荐模型 | 部署方式 | 理由 |
|---|---|---|---|
| **通用意图识别、对话、RAG问答** | DeepSeek-V3 / Kimi k1.5 / 通义千问2.5-Max | 公有云API | 中文能力强、支持长上下文、Function Calling成熟、成本低 |
| **设备/风险等级精确抽取（结构化输出）** | DeepSeek-V3 / Qwen2.5-72B-Instruct | 私有部署（单机A100/L20可跑） | 对结构化JSON输出稳定性要求高，私有化后延迟更低 |
| **语音转文字（STT）** | 科大讯飞星火语音 / 阿里云Paraformer | 公有云API或边缘盒子 | 中文方言识别准确率行业领先，支持化工领域术语定制 |
| **图片理解（拍照识别泄漏/锈蚀）** | 通义千问-VL / Kimi-Vision / DeepSeek-VL | 公有云API | 多模态能力强，可识别工业现场图片中的异常 |

### 2.3 模型调用架构

```
用户请求
   │
   ▼
┌─────────────────────────────────────┐
│         AI Gateway（路由层）          │
│  - 意图分类：调用轻量模型或规则引擎      │
│  - 流量控制：按租户/用户限流            │
│  - 缓存：常见意图结果本地缓存            │
│  - Fallback：主模型超时切备用模型        │
└─────────────────────────────────────┘
   │
   ├─ 简单意图 ──→ 小参数模型 / 规则引擎（低延迟、低成本）
   ├─ 复杂对话 ──→ DeepSeek-V3 / Kimi（高质量）
   ├─ 结构化抽取 ─→ 私有部署大模型（数据安全）
   └─ 图片理解 ───→ Qwen-VL / Kimi-Vision（多模态）
```

### 2.4 成本控制策略
- **Prompt缓存**：对设备同义词库、SOP文档等静态上下文做前缀缓存（如Kimi的Context Caching、DeepSeek的Prefix Caching）
- **意图缓存**：将常见查询（如"查本周隐患"）的意图解析结果缓存到Redis，命中率可达60%+
- **模型蒸馏**：用私有部署的大模型生成训练数据，蒸馏出7B级别的领域专用小模型，替代80%的标准化抽取任务

## 3. RAG方案设计

### 3.1 知识库内容
- 企业设备台账（含别名、位号、规格参数）
- 历史隐患库（描述、定级、整改措施、结案报告）
- 安全操作规程（SOP）
- 法规标准库（GB 30871、AQ 3013 等）
- 化学品安全技术说明书（MSDS）

### 3.2 RAG架构选型

**推荐方案：向量检索 + 图数据库混合RAG（GraphRAG-lite）**

```
用户查询（如"三号泵轴承异响"）
   │
   ▼
┌─────────────────────────────────────────┐
│         Query预处理（实体识别）           │
│  提取：三号泵 → 设备别名                  │
│       轴承异响 → 异常现象                 │
└─────────────────────────────────────────┘
   │
   ├─ 实体精确匹配 ──→ 图数据库（Neo4j/NebulaGraph）
   │                    快速定位：三号泵 = P-203
   │                    关联查询：P-203 的历史异常、同型号泵
   │
   └─ 语义相似检索 ──→ 向量数据库（Milvus / Qdrant / 阿里云向量检索服务）
                        检索：历史隐患中"轴承异响"的相似案例
                        检索：SOP中"循环泵检修"相关段落
   │
   ▼
┌─────────────────────────────────────────┐
│         重排序（Reranker）               │
│  使用 BGE-Reranker-v2-m3 对候选文档打分  │
└─────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────┐
│         大模型生成（带引用）              │
│  输出：设备识别结果 + 风险定级依据 +      │
│        推荐措施（标注来源隐患单号）        │
└─────────────────────────────────────────┘
```

### 3.3 向量数据库对比

| 方案 | 适用场景 | 推荐度 |
|---|---|---|
| **Milvus** | 大规模、需要分布式、私有化部署 | ⭐⭐⭐⭐⭐ |
| **Qdrant** | 中小规模、Rust高性能、Docker一键部署 | ⭐⭐⭐⭐ |
| **阿里云向量检索服务** | 不想自运维、云上快速启动 | ⭐⭐⭐⭐ |
| **PostgreSQL + pgvector** | 数据量小、不想引入新组件 | ⭐⭐⭐ |

**MVP推荐**：Qdrant（单机Docker即可支撑百万级向量，运维简单）

### 3.4 文档切分策略
- **设备台账**：按设备条目整页切分（避免信息截断），元数据包含：设备编号、车间、设备类型
- **历史隐患**：按单条隐患切分，元数据包含：隐患编号、设备、风险等级、整改状态
- **SOP/法规**：按章节切分（如"动火作业分级标准"作为一个chunk），元数据包含：文档来源、条款号
- **MSDS**：按化学品条目切分，元数据包含：CAS号、危险性类别

## 4. 与传统系统的集成方式

### 4.1 集成策略：API网关 + 事件总线

不直接连接各系统的数据库，而是通过**标准化API**和**消息队列**进行集成，保证各系统的独立性和稳定性。

```
┌─────────────────────────────────────────┐
│           AI 化工智能平台                 │
│  （智能工作台 / 语音报隐患 / AI边栏）      │
└────────────────────────────────────┬────┘
                                     │ HTTP / WebSocket
                                     ▼
┌─────────────────────────────────────────┐
│           统一 API 网关                   │
│  - 认证鉴权（OAuth2 / JWT）               │
│  - 限流熔断                               │
│  - 协议转换（REST ↔ SOAP/ESB）            │
└────────────────────────────────────┬────┘
                                     │
        ┌────────────┬───────────────┼───────────────┐
        ▼            ▼               ▼               ▼
   ┌─────────┐  ┌─────────┐   ┌─────────┐    ┌─────────┐
   │  EHS系统  │  │ MES/DCS │   │   ERP   │    │  HR/考勤  │
   │隐患/作业票│  │实时数据 │   │设备资产 │    │人员资质  │
   └────┬────┘  └────┬────┘   └────┬────┘    └────┬────┘
        │            │               │               │
        └────────────┴───────────────┴───────────────┘
                           │
                           ▼
                  ┌─────────────────┐
                  │   事件总线（MQ）   │
                  │  RabbitMQ / Kafka│
                  └─────────────────┘
```

### 4.2 各系统集成细节

#### EHS系统（隐患、作业票、变更管理）
- **集成方式**：REST API + 数据库视图（只读）
- **数据同步**：隐患单创建后，通过MQ推送事件给EHS系统，EHS系统消费后写入自身数据库
- **字段映射**：AI平台使用通用隐患模型，通过适配器映射到EHS系统的具体字段

#### MES/DCS（实时数据、报警）
- **集成方式**：OPC UA / MQTT / 时序数据库API（InfluxDB/TDengine）
- **数据范围**：只读，获取设备运行参数（温度、压力、流量）和报警记录
- **安全隔离**：通过网闸或数据采集网关（如KEPServerEX）单向采集，AI平台不反向控制DCS

#### ERP（设备资产、物料、成本）
- **集成方式**：SAP RFC / 用友/金蝶 OpenAPI
- **典型场景**：AI查设备采购日期、保修状态、备件库存

#### 人力资源系统（特种作业证、培训记录）
- **集成方式**：REST API 或 LDAP/AD 查询
- **典型场景**："查张三的特种作业证是否到期"

### 4.3 数据一致性保障
- **事件驱动**：AI平台创建隐患单后，发送 `HazardCreatedEvent`，EHS系统异步消费
- **补偿机制**：若EHS系统消费失败，进入死信队列，人工介入或自动重试
- **对账任务**：每日凌晨运行对账Job，比对AI平台和EHS系统的隐患单数量

## 5. 整体技术栈

### 5.1 前端技术栈

| 层级 | 技术 | 说明 |
|---|---|---|
| **框架** | Next.js 14 (App Router) | SSR/SSG支持好，SEO友好，适合企业门户 |
| **UI组件** | shadcn/ui + Tailwind CSS | 高质量企业级组件，可高度定制 |
| **状态管理** | Zustand + TanStack Query | 轻量、支持异步缓存 |
| **图表** | Recharts / ECharts | 趋势图、仪表盘 |
| **AI SDK** | Vercel AI SDK | 统一流式对话、工具调用接口 |
| **移动端** | Taro 3.x (React) 或 原生 Flutter | 推荐Taro，一套代码跑在企业微信/钉钉/独立App |

### 5.2 后端技术栈

| 层级 | 技术 | 说明 |
|---|---|---|
| **主服务** | NestJS (Node.js) 或 Spring Boot (Java) | NestJS开发快，适合AI编排；若企业已有Java团队，可选Spring Boot |
| **AI服务** | Python (FastAPI) | 专门处理大模型调用、RAG检索、图片理解 |
| **API网关** | Kong / APISIX / Spring Cloud Gateway | 统一入口、鉴权、限流 |
| **消息队列** | RabbitMQ 或 Apache Kafka | 事件总线、异步任务 |
| **缓存** | Redis | 会话、意图缓存、限流计数 |
| **向量库** | Qdrant | 知识库向量存储 |
| **图数据库** | Neo4j Community 或 阿里云图数据库 | 设备关系、隐患关联网络 |
| **关系数据库** | PostgreSQL | 主业务数据、用户权限 |
| **时序数据库** | TDengine 或 InfluxDB | 可选，存储DCS采集的时序数据 |
| **对象存储** | MinIO 或 阿里云OSS | 语音、图片文件存储 |
| **搜索引擎** | Elasticsearch | 可选，用于全文检索历史隐患 |

### 5.3 AI/算法技术栈

| 任务 | 技术 |
|---|---|
| **向量Embedding** | BGE-M3（中英多语言，开源可私有化） |
| **重排序** | BGE-Reranker-v2-m3 |
| **大模型API** | DeepSeek API / 阿里云百炼 / Moonshot API |
| **私有化部署** | vLLM / Ollama / Xinference |
| **语音转文字** | 讯飞语音听写API / 阿里云Paraformer |
| **图片理解** | Qwen-VL-Max / Kimi-Vision |
| **Workflow编排** | LangChain / LangGraph（复杂多步Agent） |

### 6.3 私有化部署注意事项
- **大模型**：若企业要求数据不出厂，在机房部署 2~4 台 GPU 服务器（A100/L20/H20），运行 vLLM + DeepSeek/Qwen
- **网络隔离**：DCS数据采集通过单向网闸，AI平台部署在办公网DMZ区
- **备份策略**：PostgreSQL每日全量+WAL增量；MinIO启用多副本；语音图片冷存归档到磁带库

## 7. 关键技术决策记录（ADR）

### ADR-01：为什么选择 Next.js 而非 Vue3？
- Next.js 的 App Router 对服务端渲染和流式AI响应（SSE）支持更成熟
- Vercel AI SDK 与 React 生态深度集成，开发流式对话体验更简单
- 若团队熟悉 Vue，可用 Nuxt 3 替代，但 AI SDK 支持稍弱

### ADR-02：为什么选择 Qdrant 而非 Milvus？
- MVP阶段数据量预计 < 100万条向量，Qdrant 单机性能足够
- Qdrant Docker 镜像更小（~100MB），启动更快，运维更简单
- 若未来扩展到多租户、千万级向量，可平滑迁移到 Milvus

### ADR-03：为什么用 Python FastAPI 做独立的 AI 服务？
- Python 生态在 LLM、RAG、CV 领域有绝对优势（LangChain、Transformers、Pillow）
- 将 AI 服务与主业务服务解耦，便于独立扩缩容、模型热更新、A/B测试
- 主业务服务用 NestJS/Java，通过 gRPC/HTTP 与 AI 服务通信

### ADR-04：为什么不直接用通用大模型做设备识别，而要加图数据库？
- 化工企业中"三号泵"这类别名非常常见，纯语义匹配容易歧义
- 图数据库能基于"用户所在车间→车间拥有的泵→泵的别名"进行精确推理，准确率从 75% 提升到 93%
- 图数据库还能提供可解释性："推荐 P-203 是因为三车间只有一台循环泵的别名是'三号泵'"

## 8. 推荐的开源/商业组合（按预算分档）

### 经济型（< 20万/年预算）
- 大模型：DeepSeek API（按量付费，价格便宜）
- 向量库：Qdrant 单机Docker
- 图数据库：Neo4j Community
- 服务器：2台 16核64G 云服务器
- 存储：阿里云OSS标准版

### 标准型（20~50万/年预算）
- 大模型：阿里云百炼 Qwen-Max + 私有化部署 Qwen-72B
- 向量库：阿里云向量检索服务 或 Milvus 集群
- 图数据库：阿里云图数据库
- 服务器：K8s 容器集群（4~6节点）
- 存储：MinIO 集群 + 阿里云OSS归档

### 豪华型（> 50万/年预算，数据不出厂）
- 大模型：本地部署 DeepSeek-V3 + 蒸馏小模型（2台 8卡A100）
- 向量库：Milvus 分布式集群
- 图数据库：NebulaGraph 企业版
- 服务器：私有云 K8s + GPU 集群
- 存储：MinIO 企业版 + 磁带库冷备
