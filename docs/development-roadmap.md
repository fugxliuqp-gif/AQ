> **核心策略**：先夯实管理后台基础功能，再扩展用户端业务；Agent 能力逐步嵌入，不做一步到位的大而全。

## 二、总体开发策略

### 2.1 分阶段推进

```
Phase 1：管理后台基础功能闭环（当前 → 2~3 周）
    └─ 超管运营能力完整化
    └─ 租户自治能力完整化
    └─ 审计日志与运营数据打通

Phase 2：数据底座与文档解析 Agent（3~5 周）
    └─ 动态 Schema 设计
    └─ 文档上传、OCR、LLM 抽取
    └─ 抽取结果确认页
    └─ 企业安全生产画像报告

Phase 3：管理类 Agent 嵌入（5~7 周）
    └─ 企业账号管理 Agent
    └─ 角色权限管理 Agent
    └─ 企业续费管理 Agent
    └─ 审批流程配置 Agent（轻量引擎）

Phase 4：用户端（L2 作业层）打通（7~9 周）
    └─ 按角色动态渲染 Agent 入口
    └─ 历史会话、收藏 Prompt
    └─ 生产业务 Agent（隐患排查、特种作业票等）
    └─ 对接真实 LLM Gateway
```

### 2.2 为什么先做管理后台？

根据需求文档，系统采用 **"平台统一管控 + 企业自治授权 + 角色分级使用"** 的三层模式：
- 如果平台无法入驻企业、无法授权模块、无法分配角色，用户端将没有"可使用的 Agent"。
- 管理后台是用户端的**前置依赖**。
- 管理类 Agent（4 个核心 Agent）的超管/企业管理员使用场景，天然落在管理后台中。

## 四、Phase 2：数据底座——智能文档解析 Agent

> 依据需求确认纪要第 20.1~20.3 条，这是 **MVP 阶段优先落地的核心能力**，是后续所有生产业务 Agent 的数据基础。

### 4.1 目标
让企业通过**上传文档**（PDF/Word/Excel/图片），自动解析并建立安全生产数据底座：
- 企业基础信息
- 空间分布信息
- 区域信息
- 生产工艺信息
- 生产物料信息
- 设备设施信息
- 特种设备信息

### 4.2 技术任务

#### 数据库层
- [ ] 设计动态 Schema 物理表（核心字段固定 + `JSONB` 扩展列）
  - `enterprise_profiles`（企业基础信息）
  - `enterprise_spaces`（厂区/建筑/区域）
  - `enterprise_processes`（工艺信息）
  - `enterprise_materials`（物料信息）
  - `enterprise_equipments`（设备设施）
  - `enterprise_special_equipments`（特种设备）
- [ ] 创建 `tenant_schema_metadata` 元数据表
- [ ] 所有业务表必须含 `tenant_id` 字段

#### 后端
- [ ] 文档上传 API（支持多文件、ZIP 包）
- [ ] 文档存储（MinIO / 本地存储）
- [ ] 文档解析队列（Bull/Redis 或 RabbitMQ，异步处理）
- [ ] OCR 接入（可选：PaddleOCR / 阿里云文字识别）
- [ ] LLM 抽取 Prompt 模板库（按文档类型分类）
- [ ] 字段扩展审批流 API（推送给企业管理员）
- [ ] 画像报告生成 API（基于已汇总数据）

#### 前端
- [ ] **文档上传中心**：拖拽上传、进度展示、解析状态追踪
- [ ] **抽取结果确认页**（需求强制要求）：
  - 左侧：原始文档预览（高亮对应段落）
  - 右侧：结构化字段表单
  - 低置信度字段红色高亮
  - 支持编辑、删除、补充字段
  - 确认入库 / 退回重解析
- [ ] 企业画像报告查看页

### 4.3 与需求确认纪要的对应

| 确认项 | Phase 2 交付物 |
|---|---|
| 20.1 文档解析聚焦 7 类基础信息 | 6 张动态 Schema 表 + 上传中心 + 解析引擎 |
| 20.2 字段扩展审批下放企业管理员 | `tenant_schema_metadata` + 审批待办 API |

## 五、Phase 3：管理类 Agent 嵌入

### 5.1 策略：不做独立的 Agent 页面，而是**在管理后台中嵌入 AI 边栏**

这种方式最符合需求中定义的 **"融合模式（C）"**：
- 管理员在熟悉的表格/表单页中操作
- 遇到复杂或重复任务时，通过右侧 AI 边栏用自然语言完成

### 5.2 4 个管理类 Agent 的落地方式

| Agent | 嵌入位置 | 首期实现能力 |
|---|---|---|
| **企业账号管理 Agent** | 超管后台 - 租户管理页右侧边栏 | "帮我创建东海化工租户""查下周到期企业""批量导入这些企业" |
| **角色权限管理 Agent** | 租户后台 - 角色管理/用户管理页右侧边栏 | "给安全部所有人加上语音报隐患权限""张三有哪些权限？" |
| **企业续费管理 Agent** | 超管后台 - 租户管理页 / 运营看板右侧边栏 | "给东海化工续费一年""查本月应收续费" |
| **审批流程配置 Agent** | 租户后台 - 新增"审批管理"页面，带 AI 边栏 | 自然语言 → 生成轻量审批流 JSON → 前端 React-Flow 渲染 |

### 5.3 技术任务

- [ ] 封装通用 AI 边栏组件（可复用于任何管理后台页面）
- [ ] Intent Router（意图路由）：识别"创建/查询/变更/配置"四类管理意图
- [ ] Named SQL / 预设 API 调用模板（安全可控）
- [ ] 对接第一家大模型 API（DeepSeek 或 通义千问）
- [ ] 审批流轻量引擎：
  - 数据表：`workflows`、`workflow_nodes`、`workflow_instances`
  - 前端：React-Flow 画布 + 节点属性配置面板
  - 后端：基于状态机的流程执行器

## 六、Phase 4：用户端（L2 作业层）打通

### 6.1 前置条件
- 管理后台已能完整入驻企业、授权模块、分配角色
- 文档解析 Agent 已建立部分企业的数据底座

### 6.2 目标
让普通人员按角色权限使用 Agent，完成一线安全生产任务。

### 6.3 功能清单

| 功能 | 说明 |
|---|---|
| **登录与鉴权** | 对接后端 JWT，登录后根据 `modules` 动态渲染工作台入口 |
| **工作台** | 只展示当前用户有权限的 Agent/模块入口卡片 |
| **AI 对话台** | 对接真实 SSE 流式对话，支持历史会话列表、会话标题自动生成 |
| **语音报隐患** | 从原型升级为真实功能：录音上传 → STT → AI 解析 → 草稿确认 → 提交 |
| **历史记录** | 查询自己的会话、操作、上报记录 |
| **收藏 Prompt** | 收藏常用指令，快速发起对话 |

### 6.4 生产业务 Agent（按优先级）

| Agent | 优先级 | 依赖 |
|---|---|---|
| **隐患排查治理 Agent** | P1 | 文档解析底座（企业信息、设备台账） |
| **特种作业票管理 Agent** | P1 | 审批流引擎 + 企业数据底座 |
| **安全培训与考试 Agent** | P2 | 知识库 RAG |
| **应急指挥辅助 Agent** | P2 | 应急预案文档已解析入库 |
| **法规标准符合性审计 Agent** | P3 | 法规知识库 |
| **职业健康管理 Agent** | P3 | 体检/防护用品数据 |

## 七、关键技术决策与约束

### 7.1 已确认的需求约束
（来自需求确认纪要）

| 约束项 | 设计影响 |
|---|---|
| 移动端 MVP 不做 | 前端优先 PC 响应式，API 保持设备无关 |
| 私有化仅技术预留 | 架构不绑死云服务，但商务上不主推 |
| 现阶段不与 DCS/MES/ERP API 对接 | 以文档上传为唯一数据入口，后续预留标准导入模板 |
| 字段扩展审批下放企业管理员 | `tenant_schema_metadata` 的审批状态由 `tenant_admin` 操作 |
| 必须支持智能抽取结果确认页 | 前端投入重点，需做成通用可复用组件 |

### 7.2 技术预留清单

为确保未来可平滑扩展，以下能力需在架构层预留接口：

- **LLM Gateway**：模型无关设计，支持 DeepSeek / Qwen / 私有化模型的切换
- **RAG 知识库**：向量数据库接口预留（当前可用 PG 的 `jsonb` 字段占位）
- **消息推送**：短信/邮件/Webhook 三种通道抽象
- **计费埋点**：模块调用、文档解析页数、Agent 对话次数的日志记录点
- **多环境部署**：配置文件化，支持 SaaS 公有云 / 私有化混合部署

## 九、风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 文档解析 Agent 的 LLM 抽取准确率不足 | 高 | MVP 阶段强制人工复核页兜底，不追求全自动 |
| 动态 Schema 查询性能差 | 中 | 先用 JSONB + GIN 索引，数据量大时再迁移到分表 |
| 前端组件库（shadcn/ui）缺少复杂组件（如流程设计器） | 中 | 审批流设计器引入 React-Flow，不硬套 shadcn |
| 大模型 API 成本超预算 | 中 | 通过 Prompt 缓存、限流、套餐配额控制消耗 |
| 化工企业接受 SaaS 模式意愿低 | 中 | 技术方案完全支持私有化迁移，保留商务灵活性 |

## 十、总结

> **当前阶段：Phase 1 进行中，RBAC 角色体系已补齐。**

**下一步最优先动作**：
1. **Week 1 内完成管理后台前后端 API 的全面对接**（租户创建、角色/用户 CRUD）。
2. **Week 2 完成审计日志和运营看板数据化**。
3. 只有当企业管理后台能完整跑通"入驻 → 授权 → 建角色 → 加用户 → 分模块"闭环后，再进入 **Phase 2（文档解析 Agent）**。
