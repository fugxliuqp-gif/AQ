# 需求吻合度分析报告

**对比基准**：`/home/fu/需求/安全生产agent线上.md`（V0.3，2025-04-17）  
**当前项目**：`test2`（截至 2025-04-17）  
**分析维度**：架构契合度、功能覆盖度、实现深度、偏差与风险

---

## 一、总体结论

当前项目已完成的底层脚手架（**SaaS 多租户架构 + 权限体系 + 管理后台前端**）与需求文档的 **L0 平台层和 L1 企业管理层底座** 高度吻合。然而，需求中定义的 **4 个核心管理类 Agent**、**智能文档解析 Agent**、**审批流程引擎**、**完整的 RBAC 角色体系** 等关键能力尚未开始构建。普通人员（L2 作业层）的用户端目前仅有静态原型页面，未与后端打通。

**一句话概括**：**地基打得对且扎实，但地上建筑（Agent 能力、业务功能、角色体系）尚未起楼。**

## 三、部分吻合/需深化项（已有底座，但未达需求深度）

### 3.1 RBAC 角色体系（中度偏差）

**需求定义**：
- 支持企业管理员创建**自定义角色**（如"安全总监""班组长""操作工"）。
- 角色绑定 **Agent 功能权限**（细粒度到每个 Agent 或子功能）。
- 支持**一人多角色**，按角色批量授权/回收权限。

**当前实现**：
- 用户表中有固定 `role` 字段枚举（`super_admin` / `tenant_admin` / `tenant_user` / `auditor`）。
- 权限通过 `user_module_permissions` **直接绑定到用户**，没有独立的 `roles` 表和 `role_permissions` 表。
- 企业管理员无法创建自定义角色，只能逐个用户分配模块。

**影响**：
- 无法满足"给安全部所有人都加上隐患排查 Agent 使用权"这类批量授权场景。
- 缺少角色级权限的灵活性和可维护性。

**建议**：
补充 `roles` 和 `role_permissions` 表，将 `user_module_permissions` 改为通过角色间接授权（或直接支持用户-角色-权限三层模型）。

---

### 3.2 4 个核心管理类 Agent（尚未实现）

需求文档第 10 章明确列出 4 个管理类 Agent，当前均**只有传统页面底座，没有 Agent 化交互能力**：

| Agent | 需求能力 | 当前状态 |
|---|---|---|
| **企业账号管理 Agent** | 通过自然语言创建/查询/变更企业账号、批量导入、生成报告 | 仅有租户管理表格页面和 CRUD API，无对话式 Agent |
| **角色权限管理 Agent** | 自然语言解析权限意图、批量授权回收、权限审计问答 | 仅有用户列表和模块开关，无角色体系和 Agent 交互 |
| **企业续费管理 Agent** | 到期预警、对话式续费、历史账单查询、自动降级恢复 | 有租户有效期字段和自动校验，但无续费 Agent 和账单模块 |
| **企业审批流程配置 Agent** | 自然语言配置多级审批流、条件分支、流程模拟调试 | 完全空白，无审批流引擎和相关数据表 |

**结论**：这 4 个 Agent 是需求的核心差异化能力，但目前仍是"待建项"。

### 3.3 运营审计与日志（仅有表，无业务逻辑）

**需求定义**：
- 全局操作日志（超管授权变更、企业停用等）。
- 异常登录/调用告警。
- 数据统计仪表盘（租户增长、Agent 调用量、活跃用户分布）。

**当前实现**：
- 数据库有 `login_logs` 和 `operation_logs` 表。
- 但**后端没有写入逻辑**（除登录接口外未记录日志），前端也没有审计查看页面。
- 运营看板的数据目前是静态 Mock 数据。

## 四、未覆盖项（完全空白，需在后续迭代中补充）

### 4.1 智能文档解析与数据汇总 Agent（MVP 优先级最高）

根据需求确认纪要（第 20.1 条），这是 **MVP 阶段优先落地的生产业务 Agent**，当前完全未涉及：

- ❌ 文档上传（PDF/Word/Excel/图片）
- ❌ OCR / 版面分析 / LLM 抽取
- ❌ 动态 Schema 数据表（核心字段固定 + `JSONB` 扩展列）
- ❌ `tenant_schema_metadata` 元数据表
- ❌ 字段扩展审批流程（下放给企业管理员）
- ❌ **智能抽取结果确认页**（需求强制要求，第 20.3 条）
- ❌ 企业安全生产画像报告自动生成

**影响**：这是需求中定义的"数据底座"，若不优先构建，其他生产业务 Agent（如隐患排查）将缺乏数据上下文支持。

### 4.2 审批流程引擎（完全空白）

需求中的第 4 个管理类 Agent 依赖审批流引擎，当前：
- ❌ 无工作流相关数据表（`workflows`、`workflow_nodes`、`workflow_instances` 等）
- ❌ 无 BPMN 或轻量状态机引擎
- ❌ 无流程设计器前端页面

### 4.3 普通人员工作台（L2 作业层）

当前前端有原型页面（`/chat`、`/mobile-hazard`），但：
- ❌ 未与后端 API 打通
- ❌ 没有"仅展示当前角色有权限的 Agent 入口"的动态逻辑
- ❌ 没有历史会话列表、收藏 Prompt、个人中心等完整功能
- ❌ 没有 Agent 调度框架（LangChain/LangGraph）

### 4.4 LLM Gateway 与 Agent 编排引擎（完全空白）

- ❌ 无 LLM Gateway 实现
- ❌ 无 Intent Router（意图路由）
- ❌ 无 Memory 模块
- ❌ 无 Tool Calling 封装
- ❌ 无向量数据库 / RAG 知识库接入

## 五、具体偏差分析

### 偏差 1：权限模型从"角色级"退化为"用户级"
**需求**：角色 → 权限 → 用户（支持自定义角色、批量授权）  
**当前**：用户 → 权限（直接绑定模块，无角色中间层）  
**风险**：企业规模稍大（如 300 人）时，权限管理将不可维护。

### 偏差 2：Agent 概念尚未产品化
**需求**：所有功能以"Agent"形态呈现（对话式、智能引导、语义导航）。  
**当前**：功能以"模块/页面"形态呈现（传统表格、表单、按钮）。  
**风险**：产品缺乏差异化竞争力，与市面上传统 SaaS 无异。

### 偏差 3：动态 Schema 设计缺失
**需求**：文档解析后的字段可动态扩展，且由企业管理员审批。  
**当前**：数据库是固定 Schema，无 `JSONB` 扩展列设计。  
**风险**：无法满足不同企业文档格式差异极大的现实需求。

## 八、总结

| 维度 | 评分（5分制） | 说明 |
|---|---|---|
| **架构契合度** | 4.5/5 | SaaS 三层架构、租户隔离、JWT 认证与需求完全对齐 |
| **功能覆盖度** | 2.5/5 | 管理后台底座完成约 60%，核心 Agent 和文档解析尚未开始 |
| **实现深度** | 2/5 | 前端多为静态原型，后端缺少角色体系、日志写入、Agent 引擎 |
| **与 MVP 需求匹配度** | 2/5 | MVP 优先级最高的"智能文档解析 Agent"完全空白，这是最大缺口 |

**最终建议**：
> 当前项目非常适合作为**继续深化的基座**。建议暂停新增页面原型，优先完成 **RBAC 角色体系** 和 **动态 Schema + 文档解析 Agent** 的数据层设计，这两项是后续所有业务 Agent 能够跑起来的前提条件。
